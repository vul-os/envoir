§0 the problem

Email was never really yours

Every inbox today is a courtesy, extended by whoever's server it lives on. DMTAP starts from a different premise: the identity is yours before any provider is involved.

01

Your address is a lease

It lives on someone else's domain, on someone else's server — and it can be suspended, sold, or shut down without your consent.

02

The provider can read it

Centralized inboxes mean centralized reading, whatever the assurances. The operator of the mailbox is, by construction, able to see the mail.

03

Metadata is the whole graph

Even where content is protected, who emailed whom, when, and how often is usually sitting in the clear — a complete social graph, timestamped forever.

§0.3 – §0.4 how it works

Three layers of indirection free your address from any server

A name resolves to a key once. After that, contacts route by key through the mesh — no domain, registrar, or server sits in the middle of an existing relationship again.

Name · human, stable
abc@def.com
Looked up once, via DNS + key transparency (§3)
Key · durable, portable
IK: 7f2a…9c1b
The identity itself — never changes on a whim (§1)

A message between two DMTAP identities never touches a mail server. It's built, sealed, and routed hop-by-hop by the sender's own node:

$ envoir send alice@yourdomain [1] resolve alice@yourdomain → key K dns + key-transparency, §3 [2] fetch K's KeyPackages + current location mesh DHT, §4 / §5.3 [3] build MOTE — sealed-sender · MLS/HPKE-encrypted · signed §2 / §5 / §6 [4] send via mixnet, private tier — onion-routed, minutes §4 / §6 [5] ack delivered & stored on alice's node · sender retries until ack no gateway · no SMTP · no plaintext outside the two endpoints

§3 naming

Pick as many names as you like. They all point at one key.

The identity is always the keypair. A name is only a memorable pointer to it — and DMTAP defines a ladder of name forms, from zero-authority to fully human-chosen, that you can hold at the same time.

Key-name
default maple-heron-otter-cabin-river-slate-amber-quill zero authority
Fingerprint k:7f2a9c1b… none
Handle @thisismyname thin directory
name@domain me@yourdomain your DNS
Self-sovereign thisismyname · name-chain chain ordering

Every identity gets its key-name the instant a key is generated — no directory, no registration, no waiting. It's 8 words drawn from an ~1,024-word list (80 bits), with a checksum folded into the last word so a mistyped or mistyped name fails closed instead of quietly resolving to someone else's key. Want it adversary-proof forever instead of just memorable? Use the 12-word form.

§5 · §13 one identity

Mail, chat, files, and login — one key, no central provider

MLS groups carry 1:1 messages, chat, and file folders on the same substrate as mail. The same key that receives your mail signs you into the web.

Mail

Async, store-and-forward, delivered even while your node sleeps — with a gateway bridge into the legacy SMTP world.

Chat

1:1 and group messaging over MLS, with async join via KeyPackages — no one has to be online at the same time.

Files

Content-addressed, encrypted chunks shared through the same MLS groups as your conversations — no separate drive to trust.

Sign-in

An origin-bound challenge, signed by your key — phishing-resistant like a passkey, because a passkey is what gates it.

On any site: “Sign in with Envoir” → enter alice@yourdomain → approve on your device. No password created, no bearer token to steal. DMTAP-Auth, §13 →

§6 privacy, honestly

Metadata-private by architecture, not by promise

Content privacy is table stakes. DMTAP also treats the social graph itself — who talks to whom, when, how much — as something to protect.

  • Sealed sender

    Intermediaries relay a MOTE without ever learning who sent it.

  • Mixnet + cover traffic

    Onion-routed hops, mixing delays, and padding hide timing and size — email's tolerance for latency is what makes this affordable.

  • Push, not poll

    Your always-on node receives through the mixnet — there's no store-and-poll step for an observer to time.

  • Private lookups

    Name→key resolution is routed through the mixnet too, so the directory never learns who's asking about whom.

honest boundary

DMTAP targets a global passive adversary — an observer who can watch traffic but not actively manipulate every hop.

It does not claim perfect resistance to a global active adversary with unlimited resources. That's a deliberately drawn line, not a marketing gap: every privacy guarantee here is one we can defend on the spec's own terms.

Messages can choose private (full mixnet, minutes) or fast (fewer hops, seconds, less metadata protection). Default is private.

§12 operators

Free to self-host, forever. Paid is optional — for not running it yourself.

Envoir is MIT-licensed end to end: protocol, node, gateway, and client. Nothing that matters is held back for a paid tier.

Self-host

$0 · your hardware
  • Run the node on a Pi, NAS, old laptop, or $2 VPS
  • Full protocol: mesh, mixnet, MLS, DMTAP-Auth — nothing gated
  • Unlimited by default — the operator seam's self-host implementation is a no-op
  • MIT license, auditable top to bottom

Hosted operator

optional · e.g. Envoir Cloud
  • Someone else runs the always-on node for you
  • Billing, quotas, and multi-tenant management
  • Same protocol, same client, same privacy guarantees
  • Never a crippled feature set — the paid thing is only running it

Both modes are the same code, talking through one documented contract — the dmtap-seam: metering, provisioning, policy, and gateway authorization. Self-host gets the unlimited defaults; a hosted operator implements the same seam without forking or gating the protocol.
